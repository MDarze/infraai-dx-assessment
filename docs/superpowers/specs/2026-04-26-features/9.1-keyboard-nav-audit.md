# 9.1 — Keyboard navigation audit

**Priority:** P2 · **Effort:** S · **Theme:** Accessibility

## Problem
Survey already has good keyboard shortcuts (1–9, Enter, Esc). Other pages are unaudited:
- Start: form fields are focusable (browser default), but tab order may not match visual order
- Results: action buttons in footer; modals (if added by 2.2) need focus traps
- AppHeader: toggles tabbable but order not guaranteed

For KSA gov tenders, accessibility is increasingly a requirement.

## Solution
Walk every page with keyboard only. Verify:
- Tab order matches visual order in both RTL and LTR
- Focus ring is always visible (using `focus-visible:ring-2 focus-visible:ring-brand`, already in Button primitive — verify everywhere)
- Skip-to-content link at the very top (jumps Tab past header)
- Esc closes modals (when modals exist)
- Arrow keys work in segmented controls (LocaleToggle, etc.)
- No keyboard traps (especially in custom inputs)

## Acceptance criteria
- [ ] Documented audit checklist with per-page passes
- [ ] All findings fixed
- [ ] Skip-to-content link present
- [ ] Focus-visible ring on every interactive element
- [ ] axe-core or lighthouse a11y audit clean

## Files
- Audit doc (this spec, or new in `docs/`)
- Various component files for fixes

## Out of scope
- Full screen-reader pass (covered by 9.2)
- High-contrast theme (separate)
