# 10.1 — Error tracking (Sentry / similar)

**Priority:** P1 · **Effort:** S · **Theme:** Operational

## Problem
Right now if a user's browser throws an error mid-survey, we never know. They abandon, we have no signal, no reproduction context. Production blind spot.

## Solution
Add Sentry (free tier handles ~5K events/month) to the frontend. Capture:
- Uncaught errors
- Unhandled promise rejections
- React error boundaries (add a top-level boundary that reports + shows fallback UI)
- Network errors from `api.ts`

Tag every event with: locale, theme, route, assessmentId (if known), build version.

Source maps uploaded at build time so stack traces resolve to original code.

Alternatives: Bugsnag (similar), Rollbar (similar), or homegrown via console.error hook + backend log endpoint (cheaper but more work).

## Acceptance criteria
- [ ] Sentry SDK initialized in `main.tsx`
- [ ] React error boundary at App root with friendly fallback
- [ ] Source maps uploaded on Railway / Pages builds
- [ ] No PII in error context (don't tag email, name)
- [ ] Production-only (skip in dev to avoid noise)

## Files
- Add `@sentry/react` dep
- `src/main.tsx` — init
- `src/App.tsx` — error boundary
- CI workflow — source map upload

## Out of scope
- Performance monitoring (different SDK feature)
- Backend Sentry (separate decision in backend repo)
