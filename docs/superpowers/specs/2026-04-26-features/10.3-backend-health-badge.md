# 10.3 — Backend health badge

**Priority:** P2 · **Effort:** S · **Theme:** Operational

## Problem
When sync fails, the user can't tell if it's their network, the backend being down, or a config issue. Right now they see a generic error and no way to triage.

## Solution
On the Settings (or About) page, a small status indicator that pings `GET /api/health` (or `/`) on mount and every 30s. States:
- Green: 2xx response in <500ms
- Amber: 2xx in 500–2000ms
- Red: timeout or non-2xx

Show last-checked timestamp. If red, an explanatory line: "The backend appears to be unreachable. Your data is saved locally and will sync when connection returns."

If 4.4 (offline queue) lands, also show queue depth here.

## Acceptance criteria
- [ ] Health badge visible on Settings/About
- [ ] Auto-refreshes every 30s while page is open
- [ ] Three states with clear visual treatment
- [ ] Failure copy actionable, not technical

## Files
- `src/pages/Settings.tsx` (or new About page)
- Backend: ensure /health endpoint exists and is fast (likely already does)

## Out of scope
- Status page hosting (UptimeRobot, etc.) — separate
- Per-endpoint health (just one root probe)
